## Why

Two needs, both about making the agentic loop **driven by the OpenSpec task list, never by
hardcoded Python generation logic**:

1. **Architecture solidity for task-driven generation.** We must confirm the pipeline can
   generate TS code + TS tests **purely from the tasks it reads as an argument** (the
   `openspec:<change>` / `tasks.md` reference), with **no hardcoded Python that emits
   specific TS logic**. Today the generation routes through Ollama prompts fed by
   `openspec_loader.load_change` (which reads `proposal.md` + `tasks.md` + `specs/**`).
   This is the right shape; this proposal hardens the rule and adds an e2e that proves it.

2. **New feature: a current-timestamp helper in the generated TS.** Add a simple
   `YYYYMMDD-HHMMSS` timestamp utility (e.g. `getCurrentTimestamp()`) to the plugin —
   **generated by the loop from a task argument**, not hand-written Python. The e2e test
   passes that task as an argument and validates the generated TS actually works
   (produces a correctly-formatted `YYYYMMDD-HHMMSS` string), proving the loop produces
   correct, runnable TS from a task.

## What Changes

- **No hardcoded generation logic.** The Python MUST only read the OpenSpec change/task
  (passed as the `openspec:<change>` argument) and feed it to the LLM; it MUST NOT contain
  Python that emits specific TS feature code. Any such hardcoded branch is removed/refactored
  so generation is fully task-driven.
- **Add a timestamp task to an OpenSpec change** (e.g. `uuid-modal-agentic-generation` or a
  dedicated change) describing: "Add `getCurrentTimestamp()` returning `YYYYMMDD-HHMMSS`
  from the current date/time." The loop generates the TS from this task alone.
- **New e2e test** `test_task_driven_timestamp_e2e.py` (real Ollama call) that:
  - invokes the pipeline with the task/change as an **argument** (`openspec:<change>`),
  - asserts the generated TS exposes `getCurrentTimestamp()` and (via `build-app`/`test-app`
    or a direct unit check) that it returns a `YYYYMMDD-HHMMSS`-formatted string,
  - proves generation is driven by the task, not hardcoded Python.
- This complements `agentic-architecture-test-refactor` (architecture must support it) and
  `agentic-tests-real-logic` (e2e uses a real call).

## Capabilities

### New Capabilities
- `task-driven-ts-generation-e2e`: The pipeline generates TS code + tests **only from the
  task supplied as an argument** (no hardcoded Python feature logic); an e2e test proves a
  timestamp feature is generated correctly and runs.

### Modified Capabilities
- `agentic-architecture-test-refactor`: its architecture assessment confirms the pipeline is
  capable of fully task-driven generation (no hardcoded TS emitters).
- `agentic-tests-real-logic`: the new e2e makes a real Ollama call and validates real TS output.

## Impact

- `agents/agentics/src/*`: any hardcoded TS-emission logic removed; `openspec_loader` stays
  the single task source.
- New OpenSpec task (timestamp feature) in the target change's `tasks.md`.
- `agents/agentics/tests/integration/test_task_driven_timestamp_e2e.py`: new real-call e2e.
- Generated TS: `getCurrentTimestamp()` (`YYYYMMDD-HHMMSS`) added to the plugin via the loop.
